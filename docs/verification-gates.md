# 2단계 검증 게이트 — 설계 방법론

이 저장소의 모든 킷이 공유하는 핵심 구조. "AI 초안을 어떻게 믿을 수 있게 만드는가"에 대한 우리의 답이다.

## 문제: 생성은 쉽고 신뢰는 어렵다

LLM 초안의 실패는 두 지점에서 온다.

1. **방향 실패** — 목차·연구질문이 요구를 빗나간 채 수십 페이지가 생성된다. 발견 시점이 늦을수록 폐기 비용이 커진다.
2. **근거 실패** — 그럴듯한 문장 속에 존재하지 않는 출처, 출처와 다른 수치, 유령 인용이 섞인다. 읽어서는 못 잡는다 — 대조해야 잡힌다.

두 실패는 성격이 달라서 **하나의 검수로는 못 막는다**. 그래서 게이트가 두 번이다.

## 게이트 1 — 집필 전 설계검토 (방향 실패 차단)

- **시점**: 설계자(designer/rfp-analyst)가 목차·RQ·평가대응표를 낸 직후, 조사·집필 착수 전.
- **검수자**: reviewer 에이전트 모드1. READ-ONLY — 자기가 고치지 않고 판정만 한다.
- **판정**: GO / 조건부 / NO-GO. 점검 항목 예 —
  - 요구(RFP·연구목적) ↔ 목차 1:1 매핑에 구멍이 없는가
  - 고배점 평가항목이 충분한 분량·깊이로 커버되는가 (제안서)
  - RQ가 답할 수 있는 질문인가, 분석틀이 실행 가능한가 (연구)
  - 분량 산술이 맞는가 (장별 합계 ≠ 목표 분량 같은 모순)
- **개정 루프**: 반려·조건부면 설계자가 개정 → 재검토, **최대 2회**. 그 뒤에도 남는 항목은 "잔여 리스크"로 기록하고 진행한다(무한 차단 방지).
- **★사용자 승인**: 내부 게이트 통과 후 목차·핵심 RQ를 사용자에게 제시하고 승인받는다. 내부 게이트는 "실행 가능성"을, 이 게이트는 "의도 부합"을 본다.

## 게이트 2 — 집필 후 초안검수 (근거 실패 차단)

- **시점**: 초안 완성 직후, 변환(hwpx/docx) 전.
- **검수자**: reviewer 모드2. 위치·문제·해결을 갖춘 수정 요청을 [필수]/[권고]로 낸다.
- **검수 축** (킷별 4~5축): 논리 정합성 / 근거 충실성 / 도메인 타당성(정책·방법론·평가지표) / 정량성 / **참고문헌·출처 검증**.
- **출처 검증의 규율**:
  - 본문 인용 ↔ 참고문헌 목록 1:1 대응
  - 각 출처의 실재·접근 가능성 확인 (URL/DOI/KCI 등)
  - 인용된 수치가 출처 내용과 일치하는가
  - 검수자의 자기보고를 믿지 않는다 — 근거 범위를 실제로 열어 확인한다
- **개정 루프**: writer 반영 → 재검수, 최대 2회. 미해결은 잔여 리스크로 명시하고 사용자에게 보고한다.

## 실전 적발 기록 (익명화)

| 사례 | 게이트 | 적발 내용 |
|---|---|---|
| 200p 정책보고서 검수 | 게이트 2 | 실재하지 않는 출처 3건, 해외 통계 환산 10배 오류 1건 |
| 학술논문 완주 | 게이트 2 | 유령 인용 2건, 본문에 없는 고아 출처 1건 |
| 정책보고서 신규 완주 | 게이트 1 | 분량 산술 모순, 근거 희소 분야의 비교 편향 등 [필수] 3건 — 집필 전 해소 |

## 설계 원칙 요약

1. **검증자는 생성하지 않는다** — reviewer는 READ-ONLY. 고치는 사람과 판정하는 사람을 분리한다.
2. **게이트는 문서로 남는다** — 판정·수정요청이 파일(`02_design_review.md`, `05_draft_review.md`)로 남아 사후 감사가 가능하다.
3. **차단보다 기록** — 2회 루프 후 미해결은 진행하되 잔여 리스크로 투명하게 남긴다. 완벽주의로 파이프라인을 세우지 않는다.
4. **모르면 모른다고 쓴다** — 조사 단계부터 "근거 미확보 항목"을 명시하고, 집필은 `[보강 필요]` 태그로 이어받는다. 지어내지 않는 것이 규칙이 아니라 구조가 되게 한다.
